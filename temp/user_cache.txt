================================================================================
REDIS-КЭШ АБОНЕНТА (prefix "user") — abs-flask
Страница: GET /users/{user_id}  (пример: /users/13807)
Код: app/core/redis.py, app/api/v1/routers/users/user_info.py (SubscriberService)
================================================================================

1. КРАТКО
---------
Один абонент = один ключ Redis.
Значение — одна JSON-строка (тип Redis: STRING), не HASH.
Класс: RedisCache(prefix="user") → ключ user:{id}, TTL по умолчанию 24 часа.

Для user_id=13807 ключ:  user:13807

Абонент 13807 в тестовых заметках проекта: ФЛ (физлицо), лимитный тариф, пакет не исчерпан.
(см. Dbeaver-query.txt: «13807 - ФЛ лимитный с полным тарифов»)


2. ФОРМАТ ХРАНЕНИЯ В REDIS
---------------------------
Ключ:     user:{user_id}
          Примеры: user:13807, user:46185

Тип:      STRING (команда SET / GET)

Значение: json.dumps(dict, ensure_ascii=False)
          Сериализация в app/core/redis.py:
          - datetime/date → ISO-строка
          - Decimal, IPv4Address → str()
          - остальное нестандартное → str()

TTL:      ex=86400 секунд (24 часа) при каждой записи set()
          При чтении TTL не продлевается автоматически.

Пустой кэш: GET user:13807 → (nil)


3. ЧТО НЕ ЯВЛЯЕТСЯ ЭТИМ КЭШЕМ
------------------------------
operator:{abs_user_id}  — кэш оператора ABS (роль, full_name), middleware auth.
                        Другой префикс, другая структура, TTL 600 с.

service:{sname}         — отдельный кэш тарифа для ЮЛ; при reset-cache user для ЮЛ
                        может удаляться дополнительно.

prefix station:*        — кэш станции, не абонента.

save_in_redis() в app/utils/redis_functions.py (HSET) — универсальная утилита;
кэш абонента на профиле /users/* использует именно RedisCache (SET + JSON).


4. КОГДА СОЗДАЁТСЯ И ОБНОВЛЯЕТСЯ
---------------------------------
4.1. Первичное наполнение (полная сборка)
    GET /users/{id} → profile_page → SubscriberService.get_subscriber_data()
    Если кэша нет:
      - читает users.user, ip_group, hotspot, satellite, user_details (ФЛ), jur_client_list (ЮЛ)…
      - собирает dict через SubscriberFormatter
      - background_tasks.add_task(cache.set, user_id, result)  — запись в Redis после ответа

4.2. Чтение при наличии кэша
    - Берётся cached из Redis
    - balanse ВСЕГДА перечитывается из БД (users.user.balanse) и подставляется в ответ
    - user_status / status сверяются с БД; при расхождении — правка и cache.set()
    - для ФЛ: is_actual сверяется с user_details
    - для ЮЛ: month_start_balance* подмешиваются в ответ (из jur_monthly_bill_v2), в кэш при dirty — set

4.3. Точечные обновления (patch существующего JSON)
    - balanse          — pays/views.py (платежи, трансферы)
    - password, hashed_password — смена пароля
    - login            — смена логина
    - id_grp, station_name, gmt, hotspot*, satellite*, hub, remove_check — смена IP-группы
    - auto_renew       — toggle_autorenew (tariffs_logic.py)
    - traffic_update_hour, reset_local_time, reset_moskow_time — change_reset_hour
    - is_archive, user_status, status — архивация / разархивация
    - поля ФИО/адреса/email/mob_tel и др. — update_detailed_info (merge в кэш)
    - remain_traffic, is_low_speed, speed — только после «сброса скорости» (tariffs.py);
      в обычной сборке SubscriberService эти поля НЕ кладутся

4.4. Полный сброс ключа (delete)
    - POST /api/v1/users/{id}/reset-cache?prefix=user
    - смена белого IP (add/delete) — delete, пересборка при следующем заходе
    - freeze/unfreeze тарифа, delete planned freeze — delete + полная пересборка
    - смена станции у абонента (stations/router) — delete по user_id
    - часть операций с тарифами — delete

4.5. Важно для «единого кэша» в другом проекте
    Тариф (остаток МБ, скорость, freeze_date, sname…) на странице профиля
    НЕ хранится в user:* постоянно — подгружается TariffService отдельно.
    Исключение: после engineer-action «сброс скорости» в кэш могут попасть
    remain_traffic, is_low_speed, speed — устаревшие, если не делать delete.

6. user_status И status
-----------------------
users.user.user_status:
  1 (или другое не 2/3) → status = 'active'
  2                     → status = 'frozen'
  3                     → status = 'archived'

define_user_status() в user_info.py.
Заморозка тарифа может менять user_status в БД; кэш синхронизируется при чтении
или через delete+rebuild после freeze/unfreeze.


7. КАК СТРАНИЦА /users/13807 ИСПОЛЬЗУЕТ КЭШ
------------------------------------------
main.py profile_page:
  user_info = SubscriberService.get_subscriber_data(13807, bg_task)
    → Redis user:13807 или сборка из БД

Из user_info в шаблон уходит subset (не весь кэш):
  user: id, login, password, balanse, email, full_name, status, is_jur, white_address, …
  station: id_grp, satellite_name, station_name, hotspot, vno
  tariff: отдельно TariffService.get_user_current_tariff() — НЕ из Redis user:*

Отдельные API (статистика, чат) читают user:* в основном для login, is_jur, gmt.

11. РЕКОМЕНДАЦИИ ДЛЯ «ЕДИНОГО КЭША» В ДРУГОМ ПРОЕКТЕ
----------------------------------------------------
Если нужен один ключ на абонента по образцу abs-flask:

  a) Ключ: user:{id}, значение: JSON object (STRING), TTL 24h.

  b) Минимальный обязательный набор (ядро):
     uid, login, balanse, id_grp, gmt, hotspot, vno, station_name,
     is_jur / is_juridical, user_status, status, email, mob_tel

  c) Разделение ФЛ/ЮЛ:
     - общие поля в корне;
     - ФЛ: full_name, address, date_birth, is_actual;
     - ЮЛ: short_name, inn, … + month_start_balance*.

  d) Не смешивать в user:* тяжёлый тарифный срез (остаток МБ, freeze_date, sname) —
     в abs-flask он живёт в TariffService; иначе кэш раздувается и устаревает.

  e) balanse: либо не кэшировать (читать всегда из БД), либо обновлять при каждой
     финансовой операции — как в pays/views.py.

  f) Пароли в кэше — осознанный риск abs-flask (нужны для отображения в профиле);
     в новом проекте лучше не хранить password/hashed_password в Redis.

  g) Инвалидация: delete при структурных изменениях (станция, белый IP, тариф freeze);
     patch — при точечных (login, auto_renew, balanse).

  h) Единообразие: не использовать HSET user:{id} для тех же данных, если основной
     путь — JSON STRING (иначе два формата одного префикса).
