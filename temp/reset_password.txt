# Сброс пароля по коду оператора — инструкция для личного кабинета (ЛК)

Документ описывает, как реализовать сторону абонента: ввод 4-значного кода,
который оператор выдал через helpdesk, и переход к смене пароля.

Связанные объекты:
  - Таблица кодов:     users.password_reset_code
  - Журнал (helpdesk): logs.helpdesk_operator_log  (ЛК пишет туда по желанию или свой лог)
  - Генерация кода:    helpdesk (POST /api/v1/helpdesk/users/{id}/password-reset/generate)

---

## 1. Общая схема

1. Абонент на странице авторизации ЛК: «Забыл пароль» → «У меня есть код для восстановления».
2. Вводит 4 цифры, которые продиктовал оператор.
3. ЛК проверяет код по таблице users.password_reset_code (сравнение хэша).
4. При успехе — форма нового пароля, затем смена пароля в users.user (или radcheck).
5. Код помечается использованным (used_at, is_active = false).

Helpdesk при этом:
  - создаёт запись в password_reset_code;
  - показывает оператору открытый код (только в Redis helpdesk, не в БД);
  - в БД хранит code_hash + code_salt.

---

## 2. Почему в БД только хэш

Открытый 4-значный код в PostgreSQL не хранится намеренно:
  - при утечке БД коды нельзя использовать;
  - helpdesk показывает код оператору из Redis (ключ helpdesk_pwd_reset_display:{user_id}, TTL = срок действия).

ЛК получает код только от абонента (устно). Проверка — через хэш.

Алгоритм (должен совпадать с helpdesk, app/api/v1/routers/helpdesk/password_reset_service.py):

  salt     = случайная строка hex, 32 символа (secrets.token_hex(16))
  code     = 4 цифры, с ведущими нулями, например "0281"
  code_hash = SHA256( f"{code}:{salt}" ).hexdigest()   # 64 символа lowercase hex

Пример на Python:

  import hashlib
  import secrets

  def hash_code(code: str, salt: str) -> str:
      return hashlib.sha256(f"{code}:{salt}".encode("utf-8")).hexdigest()

  def verify_code(entered: str, salt: str, stored_hash: str) -> bool:
      entered = entered.strip()
      if len(entered) != 4 or not entered.isdigit():
          return False
      return hash_code(entered.zfill(4) if len(entered) < 4 else entered, salt) == stored_hash

Важно: при генерации в helpdesk код уже форматируется как 4 цифры:
  f"{secrets.randbelow(10_000):04d}"
Абонент может ввести "281" — нормализуйте до "0281" (zfill(4)) перед хэшированием.

## 3. Структура таблицы users.password_reset_code

| Поле                 | Назначение |
|----------------------|------------|
| id                   | PK |
| user_id              | ID абонента (users.user.id) |
| operator_id          | Кто выдал код (users.skystream_users.id) |
| code_hash            | SHA-256 hex, 64 символа |
| code_salt            | Соль для этой записи |
| created_at           | Время выдачи |
| expires_at           | Срок действия (helpdesk: +10 минут) |
| is_active            | true — код ещё можно проверять |
| used_at              | Когда абонент успешно ввёл код |
| used_ip              | IP абонента при проверке |
| deactivation_reason  | used / expired / superseded / cancelled_by_operator |
| deactivated_at       | Когда сняли активность |
| failed_attempts      | Счётчик неверных вводов (рекомендуется увеличивать в ЛК) |
| reset_type           | Способ сброса: `email`, `bot`, `operator` (helpdesk пишет `operator`) |

Ограничение: не более одного активного неиспользованного кода на user_id
  (partial unique index uq_password_reset_code_one_active_per_user).

## 4. Как найти запись для проверки

Активный код для абонента user_id:

```sql
SELECT id, user_id, code_hash, code_salt, expires_at, failed_attempts
FROM users.password_reset_code
WHERE user_id = :user_id
  AND is_active = TRUE
  AND used_at IS NULL
ORDER BY id DESC
LIMIT 1;
```

Перед проверкой в коде приложения:

1. Если записи нет — «Код не найден или устарел».
2. Если now() >= expires_at — деактивировать:
     is_active = false,
     deactivation_reason = 'expired',
     deactivated_at = now()
3. Иначе — сравнить хэш введённого кода с code_hash (см. раздел 2).

Не перебирайте все исторические записи — только одна активная на пользователя.

## 5. API личного кабинета (рекомендуемый контракт)

Реализуйте в проекте ЛК (отдельно от helpdesk). Пути примерные.

### 5.1. POST /api/lk/password-reset/verify

Тело:
```json
{
  "login": "andreysokolov",
  "code": "042817"
}
```

Или при уже авторизованной сессии «забыл пароль» — только code, user_id из контекста.

Логика:
1. Найти users.user по login (или id из сессии).
2. Загрузить активную запись password_reset_code (раздел 4).
3. Проверить срок, is_active, хэш.
4. При неверном коде:
   - UPDATE failed_attempts = failed_attempts + 1;
   - при failed_attempts >= 5 (порог на ваш выбор) — деактивировать код, reason можно не ставить или 'cancelled_by_operator';
   - вернуть 400 без указания, верный код или нет (единое сообщение).
5. При верном коде:
   - НЕ менять пароль сразу;
   - выдать одноразовый токен сброса (рекомендуется) ИЛИ пометить код used сразу.

Два подхода к сессии после проверки кода:

**Вариант A (рекомендуется): двухшаговый**
- verify возвращает short-lived token (JWT/random, 15 мин), привязанный к code_id;
- POST /api/lk/password-reset/confirm с token + new_password;
- в confirm: снова проверить token, сменить пароль, затем:
    used_at = now(), is_active = false, deactivation_reason = 'used', used_ip = client_ip.

**Вариант B: проще, но жёстче**
- verify сразу ставит used_at и is_active = false;
- редирект на страницу нового пароля с session flag reset_allowed_user_id;
- смена пароля только если flag в сессии совпадает с user_id.

Вариант A безопаснее: если абонент закрыл браузер после ввода кода, код ещё не «сгорел» до смены пароля — обсудите с продуктом. В helpdesk код живёт 10 минут; логично после успешного verify дать 10–15 мин на ввод нового пароля через token.

### 5.2. POST /api/lk/password-reset/confirm

Тело:
```json
{
  "reset_token": "...",
  "new_password": "...",
  "new_password_repeat": "..."
}
```

Логика:
1. Валидация пароля (длина, сложность — правила ЛК).
2. Смена пароля в users.user.hashed_password / radius.radcheck (как у вас принято).
3. Завершить запись кода (если ещё не used в verify).
4. Инвалидировать reset_token.
5. Опционально: закрыть сессии radius (force disconnect), если политика безопасности требует.

Ответ: 200 + сообщение «Пароль изменён, войдите с новым паролем».

## 6. Псевдокод verify (ядро)

```python
def verify_password_reset_code(session, user_id: int, entered_code: str, client_ip: str) -> str:
    """
    Возвращает reset_token при успехе.
    Raises ValueError при ошибке.
    """
    row = get_active_code_row(session, user_id)  # см. SQL выше
    if not row:
        raise ValueError("Код недействителен или истёк")

    if utcnow() >= row.expires_at:
        deactivate(row.id, "expired")
        raise ValueError("Код недействителен или истёк")

    code = entered_code.strip()
    if len(code) > 4 or not code.isdigit():
        increment_failed(row.id)
        raise ValueError("Неверный код")

    code = code.zfill(4)
    if hash_code(code, row.code_salt) != row.code_hash:
        increment_failed(row.id)
        raise ValueError("Неверный код")

    reset_token = create_reset_token(user_id=user_id, code_id=row.id, ttl_minutes=15)
    return reset_token
```

```python
def confirm_password_reset(session, reset_token: str, new_password: str, client_ip: str):
    payload = parse_reset_token(reset_token)  # user_id, code_id, exp
    row = load_code_row(payload.code_id)
    if not row or row.user_id != payload.user_id:
        raise ValueError("Сессия сброса недействительна")
    if not row.is_active or row.used_at is not None:
        raise ValueError("Код уже использован")
    if utcnow() >= row.expires_at:
        deactivate(row.id, "expired")
        raise ValueError("Время действия кода истекло")

    set_user_password(session, payload.user_id, new_password)

    session.execute(
        update(PasswordResetCode)
        .where(PasswordResetCode.id == row.id)
        .values(
            is_active=False,
            used_at=utcnow(),
            used_ip=client_ip,
            deactivation_reason="used",
            deactivated_at=utcnow(),
        )
    )
```

## 7. Состояния записи (жизненный цикл)

```
[helpdesk: generate]
    is_active=true, used_at=NULL, expires_at=now+10min

[ЛК: verify OK] → (опционально пока не used, если вариант A с token)

[ЛК: confirm password]
    is_active=false, used_at=now, deactivation_reason='used'

[expires_at прошло, verify не успели]
    is_active=false, deactivation_reason='expired'

[helpdesk: новый generate при старом активном]
    старый: deactivation_reason='superseded'
    новый: активная запись
```

## 8. Безопасность

- Не логировать введённый код и открытый код в plaintext.
- Единое сообщение об ошибке: «Неверный код или срок действия истёк».
- Rate limit по IP и login на /verify (например 10 запросов / 15 мин).
- failed_attempts + блокировка записи после N ошибок.
- reset_token — криптостойкий, одноразовый, короткий TTL.
- После смены пароля — принудительный logout других сессий ЛК (если есть).

## 9. UI личного кабинета (минимум)

1. Страница авторизации → «Забыл пароль».
2. Выбор: «У меня есть код для восстановления» / другие способы (если есть).
3. Поле ввода 4 цифры (маска, только цифры).
4. Кнопка «Продолжить» → verify.
5. Страница «Новый пароль» + подтверждение → confirm.
6. Успех → редирект на авторизацию с подсказкой войти новым паролем.

## 12. Что НЕ делает ЛК

- Не генерирует код (только helpdesk).
- Не читает Redis helpdesk_pwd_reset_display:* (там открытый код только для оператора).
- Не восстанавливает пароль без успешной проверки хэша.
- Не хранит открытый код в своих логах.

---

Важно: в ЛК при успешной проверке кода нужно выставлять used_at, is_active = false, deactivation_reason = 'used' (см. temp/reset_password.txt). Иначе helpdesk не увидит использование.
